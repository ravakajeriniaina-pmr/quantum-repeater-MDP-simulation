Correction 2 — transitions.py : Intégrer p_swap (CRITIQUE)
Fonction à modifier : transition et transition_compact
Problème : l'action SWAP est traitée comme terminale sans probabilité d'échec. Quand le swap échoue (prob 1 - p_swap), les deux liens sont consommés et le système repart de (0,0,0,0).
Changement : ajouter p_swap comme paramètre, et pour l'action SWAP retourner deux successeurs possibles :
succès   (prob p_swap)     → état terminal (épisode terminé)
échec    (prob 1-p_swap)   → (0, 0, 0, 0) avec WAIT dynamics

Correction 3 — rewards.py : Récompense en cas d'échec du swap (CRITIQUE)
Fonction à modifier : reward et reward_from_state
Problème : la récompense suppose que le swap réussit toujours. Avec p_swap < 1, la récompense espérée est p_swap * secret_fraction(w_out).
Changement :
# Avant
R(SWAP) = secret_fraction(w_out)

# Après
R(SWAP) = p_swap * secret_fraction(w_out)

Correction 4 — value_iteration.py : Intégrer p_swap dans le solveur (CRITIQUE)
Fonction à modifier : solve_mdp
Problème : le précalcul des transitions pour SWAP suppose un retour déterministe à (0,0,0,0). Avec p_swap, le successeur après SWAP est une mixture :
p_swap     → restart depuis (0,0,0,0) avec WAIT
1-p_swap   → restart depuis (0,0,0,0) avec WAIT (swap échoué, même résultat)
Note importante : dans les deux cas (succès ou échec), les deux liens sont consommés et le système repart de (0,0,0,0). La différence est uniquement dans la récompense. Donc la transition reste (0,0,0,0) mais la récompense change.
Changement : passer p_swap en paramètre de solve_mdp et l'utiliser dans le calcul de la récompense.

Correction 5 — mc_engine.py : Intégrer p_swap dans la simulation (CRITIQUE)
Fonction à modifier : run_episode
Problème : quand l'action est SWAP, le résultat est toujours un succès. Il faut tirer aléatoirement selon p_swap.
Changement :

if action == SWAP:
    w_out = swap_werner(a1, a2, w0, t_coh)

if action == SWAP:
